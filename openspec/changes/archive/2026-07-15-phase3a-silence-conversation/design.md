## 架构

```
API Bridge 消息管线
     │
     ├── api_to_worker() —— AI 输出方向
     │       │
     │       ├── 记录最后 AI 响应时间
     │       ├── conversation_flow.on_ai_response() — 检测追问计数
     │       └── 注入难度提示到系统消息
     │
     ├── worker_to_api() —— 用户输入方向
     │       │
     │       ├── 记录最后用户输入时间
     │       └── conversation_flow.on_user_response() — 更新追问状态
     │
     └── 沉默检测（被动触发）
             │
             └── 下次 AI 响应时检查: (now - last_any_message) > 60s?
                     └── YES → 发送合规退出消息 → 关闭会话
```

## 沉默检测策略

- **追踪最后消息时间**：同时追踪 AI 响应和用户输入的时间戳
- **被动触发**：检测在 `api_to_worker()` 收到下一条 AI 响应时执行。若用户沉默，无 AI 响应则检测不触发——这是 API Bridge 代理架构的固有限制。真正的主动轮询需独立后台任务
- **触发退出**：沉默 > 60s → 发送合规退出消息 → 关闭会话

## 对话流策略

- **追问检测**：AI 输出包含"吗""呢""？"等标记 → question_count++
- **强制陈述**：question_count >= 2 → 向 system_prompt 追加"下一轮请使用陈述句，不要提问"
- **难度调整**：连续无回应的有效输入 → 降级难度

## 设计决策

1. **后台沉默检测非独立线程**：在现有消息循环中嵌入超时检查，避免多线程复杂性
2. **对话流状态仅注入 Prompt 提示**：不修改 API Bridge 核心转发逻辑，通过修改 system_prompt 注入状态
